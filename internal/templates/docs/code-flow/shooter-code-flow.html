The browser hits GET /faucet/game/{roomID}. The server renders the shooter template with the faucet address embedded so the client knows where the community wallet lives.

Step 2 — WebSocket handshake

The JS client opens a WebSocket to /faucet/ws/{roomID} with query params: nick (display name, max 12 chars), team (red/blue), and address (the player's nano_ receiving address).

The server calls upgrader.Upgrade(), then reads and validates all params. Nickname is stripped of control characters. The client IP is captured via faucetClientIP() for abuse detection.

Step 3 — Player joins room

Hub.JoinRoom() creates the room if it doesn't exist, then calls Room.Join() which assigns a colour from the team palette and a random spawn position on the 25×17 grid.

Three goroutines are started concurrently:

go faucetReadPump(ctx, p) // reads WebSocket → room.inputCh

The DB records the session (playerID, address, roomID, team, IP, nick).

Step 5 — Game loop (50ms tick)

The room goroutine Room.Run() ticks every 50ms. Each tick it calls broadcastState() which serialises the full world (all players, positions, health, heal packs) and pushes it to every connected player's send channel.

The room also drains inputCh — actions queued by faucetReadPump — and dispatches them through applyInput().

  case <-ticker.C:     broadcastState()
  case in := <-inputCh: applyInput(in)
  case <-done:         return
}

Step 6 — Player sends action

The client sends a JSON action over the WebSocket. Valid actions: move, shoot, help (heal), reload.

faucetReadPump() unmarshals the message and calls room.Submit(Input{...}) which enqueues it to room.inputCh. The room loop picks it up on the next tick via applyInput().

For shoot: applyShoot() validates the shooter is alive, the target exists and is an enemy, then checks line-of-sight using Bresenham's algorithm (HasLineOfSight() in physics.go:53). It deducts 33 health from the target.

For heal: applyHelp() requires the target to be at exactly 33 health, on the same team, and adjacent (Manhattan distance ≤ 1). Restores 33 health.

Step 7 — Kill/heal confirmed, reward signal sent

If the action results in a kill (target health → 0) or a heal, the room checks for same-IP abuse: if shooter and target share the same remote IP and DisableSameIPCheck is false, the reward is blocked and a log entry is written. No punishment — just no reward.

Otherwise, a non-blocking send pushes "kill" or "heal" into the player's FaucetRewardCh (capacity 16). If the channel is full the signal is silently dropped.

select {
  case shooter.FaucetRewardCh <- "kill":
  default: // channel full — drop
}

The payout loop goroutine picks this up immediately.

Step 8 — Daily payout limit check

payoutLoop() receives the signal and calls sendReward(ctx, p, reason) with a 300-second timeout context. This is sequential per player — no concurrent sends.

Before any Nano is sent, it checks:

  • Faucet wallet is configured (FAUCET_SEED set)
  • Player provided an address
  • IP has not exceeded 20 payouts per day (UTC midnight reset)

Set FAUCET_TEST_MODE=true to bypass the limit and skip real sends during development.

Step 9 — Building the Nano send block

FaucetSender.Send() pops the pre-cached PoW, then calls nano.SendFast():

1. rpc.AccountInfo()      // get frontier, balance, representative
2. validate balance ≥ amount
3. newBalance = balance - amount
4. addressToPublicKey()   // decode player's nano_ address
5. if no cached PoW:
     rpc.GenerateWork()   // race: remote GPU vs local CPU Blake2b mining
6. stateBlockHash()       // Blake2b-256 of all block fields
7. ed25519.Sign(hash, privateKey)
8. rpc.ProcessBlock("send", block)

Reward amount: 10000000000000000000000000 raw = 0.00001 XNO per kill or heal.

Step 10 — Proof-of-Work generation

Nano requires a small PoW nonce for every block to prevent spam. GenerateWork() races two concurrent approaches and returns whichever finishes first:

  • Remote GPU peers — called via the Nano RPC work_generate action (fast, ~200ms if available)
  • Local CPU — generateWorkCPU(): pure Go Blake2b mining loop (~1–5s at send difficulty)

At startup, FaucetSender.Init() pre-computes PoW for the current wallet frontier in the background so the very first send is instant. After each send, a new pre-cache is triggered immediately for the next frontier.

Step 11 — Block submitted to the Nano network

Client.ProcessBlock() posts the signed state block to the configured Nano node (primary first, then fallback). The node validates and broadcasts it to the network.

Retry logic: 3 attempts per URL, with exponential backoff on 429 / 5xx responses. RPC timeout per attempt: 60s.

On success the confirmed block hash is returned.

Step 12 — Payout confirmed, player notified

After the block is confirmed, two things happen in parallel:

  • The DB records the payout: reason, to-address, IP, amount raw, block hash.
  • The player receives a JSON message that the browser renders as an in-game toast/banner.
{
  "type":   "faucet_reward",
  "reason": "kill",          // or "heal"
  "xno":    "0.000010",
  "earned": "0.000050"       // running session total
}

The player's FaucetEarned big.Int accumulator is updated so the running total stays correct for the session even without querying the DB.

Bot Step 1 — Bots page

Bot Step 2 — Client POSTs reward request

When the client-side bot logic detects a kill, the browser fires a POST /faucet/bots/reward with the player's address. The server immediately checks:

  • Must be POST
  • Address must be present
  • IP must not exceed 20 payouts/day

Unlike the multiplayer flow, there is no same-IP check between players — just the daily IP rate limit.

Bot Step 3 — Immediate HTTP response

Unlike the WebSocket flow where the client waits for faucet_reward, the bot endpoint responds with HTTP 200 immediately — before PoW is generated or the block is sent. This prevents the browser from hanging on a slow request.

Bot Step 4 — Background Nano send

The actual Nano send happens in a goroutine after the HTTP response has already been written. The flow is identical to the multiplayer payout path:

go func() {
  hash, err := sender.Send(ctx, address, faucetRewardRaw)
  if err != nil {
    log.Printf("bot reward send failed: %v", err)
    return
  }
  db.RecordFaucetPayout("kill", address, ip, faucetRewardRaw, hash)
}()

The 0.00001 XNO reaches the player's wallet asynchronously. If the send fails, only a server-side log entry is written — the client already received its confirmation message.
